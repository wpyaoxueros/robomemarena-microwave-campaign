# Task22 v15 Cancelled Diagnostic: Strict Hold Froze VLM Progress

## Validated Before Cancellation

- The original `35999` checkpoint loaded its built-in norm asset directly.
  The server emitted no asset fallback warning.
- The repaired Task22 stage contract passed before rollout.
- All `ORACLE_*` controls remained zero. No completed-task prompt injection,

- `pick tomato` completed the official lift stage at step `156` and started an
  EEF hold at step `177`.

## Why It Was Stopped

The strict hold-release rule correctly rejected the VLM's premature
`place tomato aside` candidate because the immediate next label was
`pour first`. Once held, however, the environment stopped advancing and the
VLM observed the same static image. From steps `177` through `1197`, it kept
emitting `place tomato aside` and never emitted `pour first`.

Continuing to the 2000-step limit could not produce a new observation or a
new VLM decision, so the job was cancelled at scheduler elapsed time `00:05:36`
instead of consuming more GPU time. No final summary or video is claimed for
this interrupted diagnostic.

## Consequence for the Next Test

The EEF gate is useful for preventing the pre-hold switch. The strict
"immediate-next-label" release condition is not suitable here because it
freezes the VLM input. The next isolated variant retains the EEF gate but
allows the VLM's own later candidate to release the hold; it still does not
write a replacement prompt.
